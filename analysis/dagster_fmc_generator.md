# VaultSpeed FMC-to-Dagster Transformation Pipeline Analysis

## Two Repos, One Deployment

This repository spans two separate projects:

| Repo | Contains | Purpose |
|------|----------|---------|
| **Repo 1** (kris.py Container) | `kris.py` + `Dockerfile` + `requirements.txt` | Builds the kris.py code-generation tool as a container image and deploys it to a container registry |
| **Repo 2** (Eliassen Dagster Project) | `vaultspeed/FMC/*.json` + `DataWranglers/` | Eliassen's Dagster project containing the VaultSpeed FMC JSON mapping definitions and the Dagster artifacts (assets, jobs, schedules, sensors, resources, custom modules) generated by the transformation |

At build time, the kris.py container runs against Eliassen's Dagster project, reading the FMC JSON files from `vaultspeed/FMC/` and generating/updating the Python files inside `DataWranglers/`.

### Multi-Tenant Distribution Model

The kris.py container is a generic tool distributed to multiple client organizations via AWS ECR. Each client gets their own Dagster project repository (Repo 2) containing their VaultSpeed FMC exports and generated artifacts, but they all pull the same kris.py container image from a shared ECR registry.

**Client onboarding** follows this process:
1. Create an IAM user in AWS console named after the client organization
2. Attach the `ECR_Pull_Access` managed policy (read-only ECR access)
3. Generate CLI access keys for the IAM user
4. Provide the access key ID and secret to the client
5. Client configures their repository's GitHub secrets/variables with the AWS credentials
6. Client can now run `conv_FMC.yml` to pull the kris.py image and generate their Dagster project

This means the kris.py container image (Repo 1) is maintained centrally, while each client's Dagster project (Repo 2) is independent. Version selection is handled via the `VERSN` input in `conv_FMC.yml`, allowing clients to pin to a specific kris.py version or use `latest`.

### kris.py Version History

| Version | Changes |
|---------|---------|
| 0.2 | Initial release — single source handling only (does not work correctly with more than 1 source) |
| 0.3 | Multisource updates — complex sensor definition for multisource, cleanup of asset flow to integrate load cycle ID into initial flow stored procedure. Does not support object-based loading. |

## How `kris.py` Transforms FMC JSON into Dagster Components

### Input: 3 Types of JSON Files Per Source

For each source system (KAN, BH1, SSC, etc.) and load type (INIT/INCR), VaultSpeed exports:

1. **`FMC_info_*.json`** — Metadata: `dag_name`, `src_name`, `flow_type` (FL=fact load, BV=business vault), `load_type` (INIT/INCR), `schedule_interval` (cron), `start_date`, `map_mtd_file_name`
2. **`mappings_*.json` / `BV_mappings_*.json`** — Ordered map of Snowflake stored procedures with dependency chains
3. **`FL_mtd_*.json` / `BV_mtd_*.json`** — Additional metadata (referenced but not directly parsed by kris.py)

### Transformation Process (kris.py)

#### Step 1: Parse FMC Info Files (line 93–107)

- Scans for all `FMC_info_*` files
- Extracts `dag_name` as the group name (lowercased, e.g. `"init_ref"`, `"incr_kan"`)
- Collects schedule intervals for INCR loads

#### Step 2: Parse Mapping Files (line 109–119)

- Loads the corresponding `mappings_*.json` or `BV_mappings_*.json`
- Each key is a stored procedure name, with `dependencies`, `map_schema`, and `input` parameters

#### Step 3: Generate `defAsset` Objects (line 122–142)

Each procedure in the mapping becomes a Dagster `@asset`. Three asset patterns are produced:

| Procedure Pattern | Asset Type | Behavior |
|---|---|---|
| `SET_FMC_MTD_*` (first proc) | **Init asset** — root of the DAG | Gets a new `LOAD_CYCLE_ID` from Snowflake sequence `DATAWRANGLERS.FMC_LOAD_ID.NEXTVAL`, calls `CALL schema.proc(dag_name, id, load_date)`, returns the cycle ID |
| Middle procedures (EXT\_\*, STG\_\*, LNA\_\*, LAS\_\*, etc.) | **Dependent assets** | Calls `CALL schema.proc()` with no args, depends on `SET_FMC_MTD` or other upstream procs |
| `FMC_UPD_RUN_STATUS_*` (last proc) | **Two assets**: `*_SCSS` (success=1) and `*_FLL` (failure=0) | Calls `CALL schema.proc(cycle_id, success_flag)` to record run outcome |

#### Step 4: Write Asset `.py` Files (line 144–163)

- Outputs to `DataWranglers/assets/{group}.py` (e.g., `init_ref.py`, `incr_kan.py`)
- Each file has standard imports and one `@asset` function per procedure

#### Step 5: Generate Jobs (line 165–225)

- For each source group, creates two `define_asset_job` definitions:
  - `{group}_job` — selects all assets in the group **except** the `FMC_UPD_RUN_STATUS_*_FLL` (failure) asset
  - `{group}_failure` — selects **only** the `FMC_UPD_RUN_STATUS_*_FLL` asset (called when load fails)

#### Step 6: Generate Sensors (line 237–305)

- **`incr_bv_generator`** — Monitors all incremental raw vault jobs (minus exclusion list: OA, ER, BV). When all monitored jobs complete successfully, triggers the `incr_bv_job` to load the Business Vault
- **Failure sensors** — One `@run_status_sensor` per job that watches for `FAILURE` status and triggers the corresponding `*_failure` job to call `FMC_UPD_RUN_STATUS` with `success_flag=0`

#### Step 7: Generate Schedules (line 308–341)

- Only for INCR loads that have a `schedule_interval` in their `FMC_info_*.json`
- Creates `ScheduleDefinition` with the cron expression and timezone (env var `TZN`, defaults to UTC; currently set to `America/New_York`)

#### Step 8: Wire Up the Project `__init__.py` (line 344–396)

- Adds asset module imports, loads assets with `load_assets_from_modules()`, and registers everything in the `Definitions()` object alongside jobs, schedules, sensors, and the Snowflake resource

## Data Vault Loading Architecture

The data flow follows the VaultSpeed Data Vault 2.0 pattern:

```
Source Systems                Raw Vault                    Business Vault
(Kantata, ADP,          ┌─────────────────┐          ┌──────────────────┐
 BrightHire, etc.)      │  Per-source job  │          │  BV job          │
                        │                 │          │                  │
 ┌──────────┐    INIT   │ SET_FMC_MTD      │          │ SET_FMC_MTD      │
 │ Source DB ├──────────►│ EXT_* (extract)  │  sensor  │ LNA_* (links)   │
 │           │    or    │ STG_* (stage)    ├─────────►│ LAS_* (sats)    │
 │           │   INCR   │ FMC_UPD_STATUS   │  trigger │ FMC_UPD_STATUS  │
 └──────────┘           └─────────────────┘          └──────────────────┘
                              3 AM EST                      7 AM EST
                              (daily)                    (sensor-based)
```

### Per-Source Execution Flow

1. `SET_FMC_MTD_FL_{LOAD}_{SRC}` — Initialize load cycle (get new ID from sequence)
2. `EXT_*` — Extract from source CDC views into staging (parallel, all depend on step 1)
3. `STG_*` — Stage/transform data (each depends on its corresponding EXT)
4. Hub/Link/Satellite loads (depend on STG steps)
5. `FMC_UPD_RUN_STATUS_*_SCSS` — Mark success (depends on all above)
6. On failure: sensor triggers `FMC_UPD_RUN_STATUS_*_FLL` — Mark failure

### Business Vault Flow

- The `incr_bv_generator` sensor monitors all raw vault INCR jobs
- Once all non-excluded sources complete successfully, it triggers `incr_bv_job`
- BV procedures (LNA\_\*, LAS\_\*) build cross-source links and derived satellites
- BV also has its own success/failure status tracking

## Example: Generated Asset File (`init_ref.py`)

Produced from `FMC_info_INIT_REF.json` + `mappings_INIT_REF.json`:

```python
@asset(group_name="init_ref", compute_kind="Execute_Procedure")
def SET_FMC_MTD_FL_INIT_RT(context, snowflake: SnowflakeResource):
    with snowflake.get_connection() as conn:
        cursor = conn.cursor()
        cursor.execute("SELECT DATAWRANGLERS.FMC_LOAD_ID.NEXTVAL;")
        newID = cursor.fetchall()
        cursor.execute(f"CALL ELIASSEN_PROC.SET_FMC_MTD_FL_INIT_RT('INIT_REF', '{newID[0][0]}','2024-12-16 01:00:00');")
    context.add_output_metadata({"latest": newID[0][0]})
    return newID[0][0]

@asset(group_name="init_ref", compute_kind="Execute_Procedure", deps=['SET_FMC_MTD_FL_INIT_RT'])
def REF_RT_REFBOARDREPORTGOALS_INIT(context, snowflake: SnowflakeResource) -> None:
    with snowflake.get_connection() as conn:
        cursor = conn.cursor()
        cursor.execute("CALL ELIASSEN_PROC.REF_RT_REFBOARDREPORTGOALS_INIT();")

# ... additional dependent assets ...

@asset(group_name="init_ref", compute_kind="Execute_Procedure",
       ins={"SET_FMC_MTD_FL_INIT_RT": AssetIn(key=AssetKey(["SET_FMC_MTD_FL_INIT_RT"]))},
       deps=['REF_RT_REFBOARDREPORTGOALS_INIT', 'REF_RT_REFCALENDAR_INIT', ...])
def FMC_UPD_RUN_STATUS_FL_RT_INIT_SCSS(context, snowflake, SET_FMC_MTD_FL_INIT_RT) -> None:
    # Marks run as SUCCESS (flag=1)
    cursor.execute(f"CALL ELIASSEN_PROC.FMC_UPD_RUN_STATUS_FL_RT('{SET_FMC_MTD_FL_INIT_RT}', '1');")

@asset(group_name="init_ref", compute_kind="Execute_Procedure", config_schema={"last": int})
def FMC_UPD_RUN_STATUS_FL_RT_INIT_FLL(context, snowflake, SET_FMC_MTD_FL_INIT_RT) -> None:
    # Marks run as FAILURE (flag=0), triggered by failure sensor
    cycle_id = context.op_config["last"]
    cursor.execute(f"CALL ELIASSEN_PROC.FMC_UPD_RUN_STATUS_FL_RT('{cycle_id}', '0');")
```

## GitHub Actions Workflows

Five workflows automate the full lifecycle: building the code-generation container, running it against FMC exports, deploying to Dagster Cloud, and provisioning Snowflake infrastructure.

### Workflow Overview

| Workflow | File | Trigger | Purpose |
|----------|------|---------|---------|
| **Build and Push Image** | `generateImage.yml` | Push to `main` | Builds the kris.py Docker image and pushes to AWS ECR with auto-incremented semver tag + `latest` |
| **Convert FMC** | `conv_FMC.yml` | Manual (`workflow_dispatch`) | Pulls the kris.py container from ECR, runs it against the repo's FMC JSON files, and commits the generated Dagster artifacts back to the repo |
| **Serverless Prod Deployment** | `deploy.yml` | Push to `main`/`master` or manual | Deploys the Dagster project to Dagster Cloud Serverless (production) |
| **Serverless Branch Deployments** | `branch_deployments.yml` | Pull request events | Deploys the Dagster project to Dagster Cloud Serverless as a branch deployment for PR review |
| **Configure Snowflake** | `configureSF.yml` | Manual (`workflow_dispatch`) | One-time setup of the `DATAWRANGLERS.FMC_LOAD_ID` Snowflake sequence used for load cycle tracking |

### Workflow Execution Order (Typical Lifecycle)

```
1. Configure Snowflake (one-time)     4. Prod Deploy (auto on push to main)
         │                                       ▲
         ▼                                       │
2. Build kris.py Image ──────────► 3. Convert FMC ──► commits to main ──┘
   (auto on push to main)            (manual trigger)
```

When new VaultSpeed FMC exports are committed to `vaultspeed/FMC/`, a developer triggers **Convert FMC** manually. The generated Dagster code is committed back to `main`, which triggers both a new container image build and a Dagster Cloud deployment.

### `generateImage.yml` — Build and Push Image

**Trigger**: Push to `main`

**What it does**: Builds the kris.py Docker image from the repo's Dockerfile and pushes it to AWS ECR with two tags: an auto-incremented semver version (e.g., `v1.2.3`) and `latest`.

**Versioning logic**:
- Queries ECR for existing image tags matching the pattern `v{CURMAJ}.{CURMIN}.*`
- `CURMAJ` and `CURMIN` are stored as GitHub repository variables, allowing manual major/minor bumps
- Patch version auto-increments: finds the highest existing patch, adds 1
- If no tags exist for the current major.minor, starts at `.0`

**Steps**:
1. Checkout code
2. Configure AWS credentials (access key from `vars`, secret key from `secrets`)
3. Query ECR for existing version tags and compute next version
4. Build Docker image with both version and `latest` tags
5. Push both tags to ECR

**Secrets & Variables Used**:

| Name | Type | Purpose |
|------|------|---------|
| `AWS_ACCESS_KEY_ID` | Variable | AWS IAM access key for ECR access |
| `AWS_SECRET_ACCESS_KEY` | Secret | AWS IAM secret key |
| `AWS_REGION` | Variable | AWS region (e.g., `us-east-1`) |
| `AWS_ACCOUNT_ID` | Variable | AWS account ID for ECR URI construction |
| `ECR_REPO` | Variable | ECR repository name |
| `CURMAJ` | Variable | Current major version number |
| `CURMIN` | Variable | Current minor version number |

### `conv_FMC.yml` — Convert FMC Files to Dagster Project

**Trigger**: Manual (`workflow_dispatch`) with 6 configurable inputs

**What it does**: Pulls the kris.py container from ECR, mounts the repo as a volume at `/dagster/`, runs the code generator, and commits any generated/modified files back to the current branch.

**Workflow Dispatch Inputs**:

| Input | Type | Default | Description |
|-------|------|---------|-------------|
| `project` | string | `DataWranglers` | Dagster project/module name (maps to `PROJECT_NAME` env var) |
| `STDLDT` | string | `True` | Whether to use FMC `start_date` as the load date for INIT loads (maps to `STLD`) |
| `ScheduleTimezone` | string | `America/New_York` | Timezone for INCR schedule definitions (maps to `TZN`) |
| `FMCN` | string | `False` | Custom FMC schema naming override (maps to `FMCN`) |
| `VERSN` | string | `latest` | kris.py container image tag to use (e.g., `v1.2.3` or `latest`) |
| `LoadType` | choice | `ALL` | Filter generation by load type: `ALL`, `INIT`, or `INCR` |

**Steps**:
1. Checkout code
2. Configure AWS credentials
3. Log in to Amazon ECR
4. Pull the specified kris.py container image version from ECR
5. Run the container with environment variables and `--load-type` CLI argument:
   ```
   docker run --rm \
     -v .:/dagster/ \
     -e BVEX=$BV_EXCLUDE \
     -e PROJECT_NAME=<project> \
     -e FMCN=<FMCN> \
     -e STLD=<STDLDT> \
     -e TZN=<ScheduleTimezone> \
     <ecr-image>:<VERSN> \
     --load-type <LoadType>
   ```
6. Check for modified files (`git status --porcelain`)
7. If changes exist: configure git, stage all, commit, rebase on origin, and push

**Container Volume Mount**: The repo root is mounted at `/dagster/`, giving kris.py access to read `vaultspeed/FMC/*.json` and write to `DataWranglers/`.

**Concurrency**: Uses `${{ github.ref }}/conv_FMC` group with `cancel-in-progress: true` to prevent overlapping generation runs on the same branch.

**Permissions**: Requires `contents: write` to push generated code back to the repo.

**Secrets & Variables Used**:

| Name | Type | Purpose |
|------|------|---------|
| `AWS_ACCESS_KEY_ID` | Variable | AWS IAM access key for ECR access |
| `AWS_SECRET_ACCESS_KEY` | Secret | AWS IAM secret key |
| `AWS_REGION` | Variable | AWS region |
| `AWS_ACCOUNT_ID` | Variable | AWS account ID for ECR URI |
| `ECR_REPO` | Variable | ECR repository name |
| `BV_EXCLUDE` | Variable | Comma-separated list of BV jobs to exclude from `incr_bv_generator` sensor monitoring |

### `deploy.yml` — Serverless Prod Deployment

**Trigger**: Push to `main`/`master` or manual `workflow_dispatch`

**What it does**: Deploys the Dagster project to Dagster Cloud Serverless as the production deployment. Supports two deployment modes: Docker-based and Python Executable (PEX).

**Deployment Modes**:
- **Docker deploy**: Parses `dagster_cloud.yaml` for code location definitions, builds a Docker image with the project code, and deploys it to Dagster Cloud
- **PEX deploy**: Builds a Python executable bundle and deploys it — a lighter-weight alternative to Docker

The prerun check (`dagster-io/dagster-cloud-action/actions/utils/prerun`) determines which mode to use based on the Dagster Cloud organization configuration.

**Dagster Cloud Configuration** (`dagster_cloud.yaml`):
```yaml
locations:
- code_source:
    module_name: DataWranglers
  location_name: DataWranglers
```

A single code location named `DataWranglers` loading the `DataWranglers` Python module.

**Environment Variables**:

| Name | Value | Purpose |
|------|-------|---------|
| `DAGSTER_CLOUD_URL` | `http://eliassen-group.dagster.cloud` | Dagster Cloud organization URL |
| `DAGSTER_CLOUD_API_TOKEN` | (secret) | API token for Dagster Cloud authentication |
| `ENABLE_FAST_DEPLOYS` | `true` | Enables Dagster's fast deploy mode for quicker iteration |
| `PYTHON_VERSION` | `3.10` | Python version for the deployment environment |
| `DAGSTER_CLOUD_FILE` | `dagster_cloud.yaml` | Path to the Dagster Cloud configuration file |

**Jobs**: Two sequential jobs:
1. `dagster_cloud_default_deploy` — Runs prerun checks and parses workspace; outputs `build_info` for Docker deploys
2. `dagster_cloud_docker_deploy` — Conditionally runs if Docker deploy is needed; uses a matrix strategy to deploy each code location in parallel

**Secrets & Variables Used**:

| Name | Type | Purpose |
|------|------|---------|
| `DAGSTER_CLOUD_API_TOKEN` | Secret | Dagster Cloud API authentication |
| `ORGANIZATION_ID` | Secret | Dagster Cloud organization identifier |
| `GITHUB_TOKEN` | Secret (built-in) | GitHub API access for deployment actions |

**Dagster Cloud Action Version**: `dagster-io/dagster-cloud-action@v1.12.3`

### `branch_deployments.yml` — Serverless Branch Deployments

**Trigger**: Pull request events (`opened`, `synchronize`, `reopened`, `closed`)

**What it does**: Identical to `deploy.yml` except it creates an isolated **branch deployment** in Dagster Cloud for PR review, using `serverless_branch_deploy` instead of `serverless_prod_deploy`. Branch deployments are automatically cleaned up when the PR is closed.

This allows developers to preview Dagster changes (new assets, modified schedules, etc.) in an isolated Dagster Cloud environment before merging to `main`.

**Secrets & Variables**: Same as `deploy.yml`.

### `configureSF.yml` — Configure Snowflake for Deployment

**Trigger**: Manual (`workflow_dispatch`) with 2 inputs

**What it does**: One-time infrastructure setup that creates the `DATAWRANGLERS` schema and `FMC_LOAD_ID` sequence in Snowflake. The sequence generates unique `LOAD_CYCLE_ID` values used by the `SET_FMC_MTD_*` assets to track each load run.

**Workflow Dispatch Inputs**:

| Input | Type | Default | Description |
|-------|------|---------|-------------|
| `startid` | number | `1` | Starting value for the sequence |
| `incrval` | number | `1` | Increment value for the sequence |

**Steps**:
1. Install Snowflake CLI (`snowflake-cli-labs`) in a Python venv
2. Configure Snowflake connection using repository variables and secrets (account, user, password, warehouse, database)
3. Check if `DATAWRANGLERS` schema exists; create it if missing
4. Check if `FMC_LOAD_ID` sequence exists; create or validate it based on current state:
   - If sequence doesn't exist: create it with the specified start/increment
   - If sequence exists and is configured correctly: no-op
   - If sequence exists but with wrong increment and next value > start: error (prevents data corruption)
   - If sequence exists with default start (1) and next value has advanced: informational message (already configured)
   - Otherwise: recreate with new values

**Safety checks**: Prevents accidental corruption of the load cycle ID sequence by refusing to lower the start value below the current next value. This protects the referential integrity of `LOAD_CYCLE_ID` values already written to the data vault.

**Secrets & Variables Used**:

| Name | Type | Purpose |
|------|------|---------|
| `SNOWFLAKE_ACCOUNT` | Variable | Snowflake account identifier (dots converted to dashes for CLI) |
| `SNOWFLAKE_USER` | Variable | Snowflake username |
| `SNOWFLAKE_PASSWORD` | Secret | Snowflake password (used only by this workflow; other Snowflake access uses PEM key auth) |
| `SNOWFLAKE_WAREHOUSE` | Variable | Snowflake warehouse |
| `SNOWFLAKE_DATABASE` | Variable | Snowflake database |
